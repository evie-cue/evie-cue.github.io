Circuit breakers are interface design

Most teams talk about circuit breakers like they’re a library feature. Add a dependency wrapper, set a threshold, ship it.

A circuit breaker is an API decision: what response you return when you no longer trust the dependency.

The breaker is the mechanism. The hard part is the contract. When the breaker opens, the system has to pick a behavior that is safe, understandable, and cheap. If you don’t choose deliberately, you’ll get whatever your defaults produce under panic.

What you’re actually designing

  • Failure semantics: do callers see 503, stale data, partial results, or a cached value?
  • Blast radius: does one noisy tenant trip the breaker for everyone, or is it scoped?
  • Load shape: does the fallback create more load (rendering, fanout, retries), or less?
  • Operator posture: can on-call predict what the product does while the graph is on fire?

The common mistake: “open” is not a plan

Teams often implement “open means fail fast”, and call it done. That’s sometimes correct. It’s also sometimes a product outage with nicer latency.

A breaker without a fallback is still valuable (it prevents resource exhaustion), but it’s not resilience unless the system has a degraded mode that users can live with.

Good fallbacks are boring and explicit

  • Stale-with-bounds: serve cached data with an age limit; expose Age or “last updated”.
  • Partial answers: omit the risky component; don’t block the whole page on a badge.
  • Static defaults: return a safe baseline (e.g., “no recommendations”) that doesn’t trigger more calls.
  • Queue for later: accept the request but complete asynchronously (only when users expect it).

Scope the breaker like you scope risk

Global breakers are easy to build and easy to regret. If one tenant or one keyspace is pathological, you want to trip locally. The right scope depends on the dependency and the threat model:

  • Per-host when failures correlate with instances.
  • Per-route / per-method when different calls have different error modes.
  • Per-tenant when fairness matters.
  • Per-region when you’d rather degrade than fail over into a bigger problem.

Make it observable (or it’s just a rumor)

If a breaker trips and you can’t tell why, you’ll either flap it manually or disable it in frustration. A usable breaker needs:

  • State: closed / open / half-open, exported as a metric.
  • Reason: which signals caused the trip (timeouts, 5xx, saturation, queue depth).
  • Impact: how many requests hit fallback, and what users saw.
  • Hysteresis: slow recovery, not instant optimism.

Reliability isn’t “never failing”. It’s failing in a way you can predict, explain, and operate. Circuit breakers are how you teach an interface to tell the truth under load.
