Operational checklists

Checklists get a bad reputation because people confuse them with paperwork. But a good checklist is not documentation. It’s a protocol.

A checklist is a control plane for humans: a small, executable sequence that makes change and recovery repeatable under stress.

Why teams fail without them

Under load, humans lose branching factor. We stop exploring options. We narrow to whatever feels plausible in the moment. That’s not a moral failure. That’s a property of operating during an incident.

In a distributed system, the equivalent failure mode is “implicit coupling.” Work that depends on invisible assumptions. When you remove a checklist, you don’t remove work. You remove the shared artifact that keeps the work aligned.

What a checklist should contain

The checklist is not a memory aid. It’s a decision compressor. It should reduce the number of judgments you have to make while things are on fire.

  • Triggers: when to run it (e.g. “p95 > 2s for 10m”, “error rate > 1%”, “deploy starts”).
  • Preconditions: what must be true before you continue (e.g. “rollback path validated”, “on-call acknowledged”).
  • Actions: small steps that can be executed by a competent engineer without additional context.
  • Stop conditions: explicit “do not proceed” criteria (e.g. “unknown blast radius”, “no canary signal”).

The checklist properties that actually matter

  • Short: if it’s long, you created a manual. Manuals don’t execute during incidents.
  • Observable: each step should produce a signal: a metric change, a diff, a log line, a ticket, a message.
  • Versioned: treat it like code. Diff it. Review it. Roll it back if you broke it.
  • Owned: “everyone owns it” means nobody fixes it. Assign ownership like you assign services.
  • Runnable without hero context: if it requires someone’s brain, it’s not a checklist. It’s a tribal spell.

Where checklists pay for themselves

The ROI shows up when the system is partially degraded. Not when it’s dead. The hard incidents are the ones where half the graphs look fine and users are still angry.

  • Deploy checklists: preflight (canary health, rollback verified), and postflight (SLO impact, saturation checks).
  • Incident checklists: establish comms, set a primary, declare scope, run first triage, start a timeline.
  • Pager handoff checklists: what the next on-call needs in 5 minutes, not what you want to explain in 30.

Checklists are an anti-fragility mechanism

The best checklist is the one that improves after every near-miss. Not because you “updated the docs,” but because you tightened the protocol. You made the failure mode harder to repeat.

If you want boring recovery, you need boring steps. Checklists are how you get there.
